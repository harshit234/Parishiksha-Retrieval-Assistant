# Chunking Diff — v1 vs v2

## What Was in v1

The original `chunks.json` (v1) contained **15 hand-crafted chunks** with a flat, minimal schema. Each chunk carried only three metadata fields: `chapter` (a coarse string like `"Motion"` or `"Force and Laws of Motion"`), `type` (a loose label such as `"definition"`, `"types"`, or `"concepts"` — with no enforced vocabulary), and `topic` (a short keyword phrase). There was no information about token length, so the retrieval layer had no way to respect LLM context-window limits when assembling a prompt. Page provenance was entirely absent, meaning a retrieved chunk could not be traced back to its location in the source textbook — a problem for citation and for debugging retrieval quality. The chunking code itself used a fixed `RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=128)` applied to a single concatenated blob of both chapters, so chapter boundaries were invisible to the splitter and cross-chapter chunks were possible. Content was unevenly distributed (15 chunks for content that now fills 24), and important sub-topics like *impulse*, *uniform circular motion*, *distance vs. displacement*, and *mass vs. weight* were either missing or collapsed into overly broad chunks.

## What Changed in v2 — and Why

The upgraded `chunks.json` (v2) expands to **24 semantically-focused chunks** and introduces four structural upgrades. **(1) Token-aware sizing:** every chunk now carries a `token_count` field (estimated as `words × 1.3`), making it trivial for the retrieval layer to budget tokens before stuffing context into a prompt — critical when using models with strict limits like Gemini's 8 k or 32 k windows. **(2) Controlled `content_type` vocabulary:** the loose `type` field is replaced by a strict seven-way enum — `definition`, `law`, `equation`, `example`, `list`, `concept`, `application` — detected automatically by regex heuristics in the new `_detect_content_type()` function. This enables query-type routing (e.g., "give me only *example* chunks for Newton's laws") and richer evaluation breakdowns. **(3) Page metadata:** each chunk records `page_start` and `page_end` (interpolated from a `SOURCE_PAGE_MAP` keyed to each source file), enabling inline citations in generated answers and making retrieval failures easier to trace to a specific page. **(4) Additional provenance fields:** `source` (filename), `chapter_number` (integer), `textbook` name, and a boolean `has_formula` flag are now first-class metadata, making downstream filtering by formula-heavy chunks (for math QA) or by chapter number straightforward without re-parsing chunk text.

---

| Dimension | v1 | v2 |
|---|---|---|
| Chunk count | 15 | 24 |
| Token count field | No | Yes (`token_count`) |
| `content_type` vocabulary | loose, 6 ad-hoc values | strict 7-value enum |
| Page metadata | No | Yes (`page_start`, `page_end`) |
| Source file tracked | No | Yes (`source`) |
| Formula flag | No | Yes (`has_formula`) |
| Chapter number (int) | No | Yes (`chapter_number`) |
| Cross-chapter chunks possible | Yes (single blob split) | No (per-file splitting) |
| Auto-generated by code | Partially | Fully (`create_chunks()`) |
